## Problema

O pipeline ML processa videos em 10 steps sequenciais. Hoje o pattern dominante nos workers e:

```python
try:
    result = analyze_X(...)
except Exception as e:
    logger.error(...)
    result = {"score": 0, "confidence": "failed", "metrics": {}}
```

Isso cria **mentira estrutural no banco**: `score=0` em `analysis_results` e indistinguivel entre:
- Pessoa realmente pontuou 0 (verdade)
- Analyzer quebrou e caiu no fallback (mentira: score nao deveria existir)
- Dependencia upstream falhou e downstream rodou com lixo (mentira em cascata)
- LLM retornou schema invalido e virou string vazia (mentira no report)

**Pior ainda** — identificado no audit:
1. **3 dos 10 workers** (facial, tonality, opening) foram modernizados pra `disponivel: bool`, mas o aggregator continuou checando `confidence != "failed"`. Resultado: esses workers **NUNCA aparecem como falhos**.
2. **Identity_analyzer mente ativamente** — retorna `score=50` quando falha.
3. **Aggregator so processa 6 de 10 workers** — os 4 novos entram via codepath paralelo no `app.py`.
4. **Report_generator aceita dict sem contrato** — score 0 de dimensao falhada vira "Pontuacao: 0/100" no prompt do Gemini. LLM gera coaching em cima de mentira.
5. **Schema retrocompat silencioso** no report (`summary` + `resumo`, `dimension_feedback` + `dimensoes`) — consumidores leem chaves diferentes, divergem sem aviso.

## Principio Central

> **Um dado no banco so pode existir se for rastreavel ate sua fonte de verdade. Senao, nao existe.**

Tres corolarios:

1. **Falha vira falha visivel.** Nunca `score=0` por fallback. Score e opcional (`int | None`).
2. **Schema e contrato.** Workers retornam `WorkerResult` validado por Pydantic. Sem dict solto.
3. **Agregacao parcial e explicita.** Se N de 10 workers falharam, `aggregated_metrics.partial_aggregation=true` + lista de falhas.

## Objetivo

Tornar **fisicamente impossivel** gravar dado mentiroso em `analysis_results`, `aggregated_metrics` e `reports`. Failure detection via tipo discriminado (Pydantic Union), nao via string check.

## Usuario e Motivacao

- **Estagio do produto:** MVP interno → piloto comercial
- **Usuario primario interno:** Bruno (PM/dev solo) — precisa confiar em metricas pra iterar modelos
- **Usuario primario externo:** colaborador se auto-avaliando — merece honestidade ("analise indisponivel" e melhor que score fake)
- **Usuario secundario:** dono/contratante visualizando agregados — decisoes de RH em cima de dado ruim e perigoso
- **Dor critica:** zero confiabilidade no banco hoje. Toda metrica pode ser mentira.

## Restricoes Operacionais

1. **Backward compatibility obrigatoria** — reports antigos (schema legacy) precisam continuar renderizando ate migration window fechar
2. **Migration sem perda de dados** — `analysis_results` ja tem milhares de rows; `score` vira nullable com cuidado
3. **Deploy coordenado worker+aggregator** — schema novo nos workers quebra aggregator antigo sem feature flag
4. **Time:** solo + Claude Code
5. **Ferramentas:** Pydantic v2 (ja no stack), Supabase Postgres, FastAPI
6. **Deadline:** sem deadline rigido, mas **bloqueia decisoes de produto** — qualquer feature nova rodando em cima do banco mentiroso vira mais debito

## Metricas de Sucesso

### Principal: zero mentira estrutural
- `SELECT COUNT(*) FROM analysis_results WHERE score = 0 AND dimension_status = 'ok'` → deve retornar **0** pos-migration
- `SELECT COUNT(*) FROM aggregated_metrics WHERE overall_score IS NOT NULL AND partial_aggregation = true` → esperado > 0 (honestidade sobre agregacao parcial)

### Secundarias
- 100% dos workers retornam `WorkerResult` validado por Pydantic (10/10)
- `report_generator` recebe tipo `AggregatedMetrics` (Pydantic), nao `dict`
- Zero codepath paralelo entre aggregator e app.py — uma unica funcao agregadora
- Feature flag `TRUTH_CONTRACT_ENABLED` permite rollback sem downtime
