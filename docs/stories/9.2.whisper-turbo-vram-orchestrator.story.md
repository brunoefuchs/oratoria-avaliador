# Story 9.2: Whisper large-v3-turbo + VRAM Orchestrator

## Status
Ready

## Size
L (~20h em 7 tasks)

## Business Value

Segunda story do Epic 9 (Wave 1 — Infrastructure). **Bloqueante para Stories 9.3 e 9.5** (Wav2Vec2-emotion + py-feat FACS) que precisam do orquestrador de VRAM para rodar no RTX 4060 8.6GB sem estourar memória.

**Duas mudanças independentes mas empacotadas atomicamente:**

1. **Whisper medium → large-v3-turbo** (ou `distil-large-v3`)
   - Melhor WER em PT-BR (~30% em tests acadêmicos)
   - Timestamps por palavra mais precisos → melhora cascata em fillers, opening, identity
   - Same VRAM footprint (~6GB, cabe em 8.6GB)
   - User confirmou uso em projeto anterior com áudio 2h+ sem issues

2. **VRAM Orchestrator** (`_model_loader.py`)
   - Context manager `ModelGPU` com load/unload explícito
   - Pipeline reorganizado: heavy models sequencial, light parallel
   - Elimina risco de OOM quando Stories 9.3 + 9.5 trouxerem modelos novos
   - Prepara terreno pra Gate 1 (hardware smoke) PASS

**Sem esta story:**
- Epic 9 Waves 2+3 ficam bloqueadas (risco OOM)
- Transcrição permanece subótima (medium em 2026 é legacy)
- Pipeline continua load-everything-at-once (bomba-relógio pra próximos upgrades)

## Acceptance Criteria

### AC1 — Whisper troca para large-v3-turbo via feature flag

```
Given variavel WHISPER_MODEL ou flag WHISPER_TURBO_ENABLED
When voice_analyzer.transcribe_audio eh chamado
Then usa "large-v3-turbo" quando flag=true (default) OU quando WHISPER_MODEL="large-v3-turbo"
And usa "medium" quando WHISPER_TURBO_ENABLED=false (rollback explicito)
And usa WHISPER_MODEL env var quando override manual
And log registra qual model foi carregado
```

### AC2 — Fallback automatico se turbo falhar

```
Given whisper.load_model("large-v3-turbo") lanca exception (download fail, VRAM OOM, etc.)
When voice_analyzer detecta a falha
Then tenta fallback para whisper.load_model("medium") com warning log
And dispara metric "whisper_turbo_fallback_triggered" pra observability
And NAO quebra o pipeline (resiliencia)
```

### AC3 — ModelGPU context manager criado

```
Given novo modulo ml-worker/workers/_model_loader.py
When import ModelGPU
Then exporta classe context manager com:
  - __enter__: torch.cuda.empty_cache() + load model via factory
  - __exit__: del model + torch.cuda.empty_cache() + gc.collect()
  - Suporte para nomes: "whisper_turbo", "whisper_medium", "wav2vec2_emotion" (9.3), "pyfeat" (9.5)
  - Telemetria: log peak VRAM antes/durante/depois
  - Thread-safe (lock para evitar concurrent loads do mesmo model)
```

### AC4 — Pipeline reorganizado em process_evaluation

```
Given app.py orquestra 13 workers hoje (todos paralelos via asyncio)
When feature flag MODEL_ORCHESTRATOR_ENABLED=true
Then workers reorganizados em 3 fases:
  Fase 1 (sequencial heavy): Whisper transcricao → unload
  Fase 2 (paralelo light): MediaPipe pose/hand/face + parselmouth (cabem juntos, ~3GB total)
  Fase 3 (sequencial novos ML): Wav2Vec2 (9.3) → py-feat (9.5) — quando ativados
And log total time, peak VRAM por fase, overhead de load/unload
And compat: fase 2 mantem asyncio paralelismo interno (nao regride perf)
```

### AC5 — VRAM check script

```
Given novo arquivo ml-worker/scripts/vram_check.py
When executado (python ml-worker/scripts/vram_check.py)
Then:
  - Carrega whisper turbo via ModelGPU + unload
  - Carrega wav2vec2 base (stub para 9.3) + unload
  - Carrega py-feat (stub para 9.5) + unload
  - Reporta peak VRAM em cada fase + peak global
  - Exit code 0 se peak <= 7.5GB (margem de 1.1GB pra MediaPipe concurrent)
  - Exit code 1 + error msg se OOM ou peak > 7.5GB
And integra com Gate 1 do Epic 9 (pre-Wave 1 merge)
```

### AC6 — WER benchmark vs medium

```
Given script ml-worker/scripts/wer_benchmark.py
When executado em 10 audios de calibracao com transcripts ground truth
Then:
  - Transcreve cada audio com whisper medium + whisper turbo
  - Calcula WER (Word Error Rate) para cada
  - Relatorio em docs/qa/wer-benchmark-story92.md com tabela:
    | audio | duration | WER medium | WER turbo | delta | wall_time_ms_medium | wall_time_ms_turbo |
  - Valida: WER turbo <= WER medium + 5% (tolerancia) em TODOS os audios
  - Meta: WER turbo < WER medium em >=7/10 audios (esperado)
```

### Decisões arquiteturais

1. **Duas feature flags separadas** — `WHISPER_TURBO_ENABLED` pode ativar turbo mesmo sem orchestrator (e vice-versa). Independência reduz risco de rollout acoplado.
2. **Fallback automatico medium** — whisper turbo tem sido estável, mas em hardware variado pode ter issues. Fallback garante pipeline nunca quebra por modelo novo.
3. **Factory pattern em MODEL_FACTORIES** — permite 9.3/9.5 adicionarem seus modelos sem tocar em orchestrator code. Extension-friendly.
4. **VRAM budget 7.5GB** — 8.6GB total - ~1.1GB pra MediaPipe workers paralelos + OS overhead. Margem defensiva.
5. **jiwer como lib WER** — academicamente padrão, API simples, MIT license. Alternativas (TorchMetrics) têm dependência pesada.
6. **Scripts em `ml-worker/scripts/`** — diretório novo, separado dos workers de produção. Pattern que escalará pra futuros tools.

### Modelo pre-baixado

`~/.cache/whisper/` já tem medium + large-v3. **turbo pode precisar download adicional** (~1.5GB). Task 1 check inclui: se flag ON e turbo ausente, download automático na primeira run via `whisper.load_model("large-v3-turbo")`.
