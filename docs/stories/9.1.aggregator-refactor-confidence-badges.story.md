# Story 9.1: Aggregator Refactor + Confidence Badges

## Status
Ready

## Size
M (~16h em 6 tasks)

## Business Value

Pré-requisito absoluto do Epic 9.0 — bloqueia todas as demais stories (9.2-9.6).

Hoje o aggregator tem uma **assimetria silenciosa**: `SCORING_DIMENSIONS` declara 10 dimensões, mas `PESOS_DEFAULT` e `PESOS_POR_CONTEXTO` só pesam 5. As outras 5 (`archetypes`, `facial`, `tonality`, `opening`, `identity`) são calculadas, salvas e exibidas — **mas não afetam overall_score**.

Consequência prática:
- Usuário vê score 72, não entende por que feedback diz "tonalidade monocórdica é seu principal problema"
- Gate 3 (ground truth Gui) vai comparar app score vs rubric humano sem saber que 5 dims estão fora do cálculo
- Epic 9 pretende promover `tonality` (via Wav2Vec2) e `facial` (via py-feat) para scoring — sem esta story, não existe estrutura para fazer isso honestamente

**Após esta story:** 6 dimensões scoring auditáveis (todas 🟢 Alta + variety como meta), 7 dimensões secundárias rotuladas honestamente, badges de confidence visíveis no report. Base limpa para ML novo entrar em 9.3/9.5.

### Decisões arquiteturais

1. **Feature flag padrão OFF** — rollback instantâneo. v0.6.0 em prod continua funcionando bit-idêntico até Gate 2 PASS.
2. **SECONDARY_DIMENSIONS separada de AUGMENTATION** — semanticamente diferente: AUGMENTATION é "consumida pelo report_generator" (uso interno), SECONDARY é "exibida como análise complementar" (uso externo).
3. **Confidence mapping hardcoded em contracts** — não em config. Confidence é propriedade do *método de medição*, não do ambiente. Mudar confidence = mudar código = code review obrigatório.
4. **Schema version bump obrigatório** — consumers (frontend, report_generator) precisam detectar schema novo pra renderizar badges.
5. **variety continua scoring** mesmo sendo 🟡 — justificativa no Epic doc: variety é meta-dim sobre outputs 🟢, herda confiança.
