## Business Value

Hoje o pipeline ML grava `score=0` no banco indistinguivel entre "pessoa pontuou 0" e "analyzer quebrou e caiu no fallback silencioso". Isso corrompe TODO o sistema downstream — reports, coaching, benchmarks viram castelo de cartas em cima de dado mentiroso.

3 dos 10 workers (facial, tonality, opening) NUNCA aparecem como falhos pro aggregator porque mudaram de schema (`disponivel: bool`) sem migrar o aggregator. Identity ativamente mente: retorna `score=50` quando falha. Resultado: usuario recebe `status=completed` com relatorio baseado em lixo, sem saber.

Esta story estabelece o **contrato universal** que GARANTE que todo dado salvo em `analysis_results` tem origem verificavel, status determinado e schema validado por Pydantic.

**Impacto direto:**
- Zero score mentiroso no banco (validavel via SQL)
- Frontend pode mostrar "analise indisponivel" honestamente
- Base de dados confiavel pra iteracao de modelos (8.3 dashboards)
- Foundation pra Stories 8.2, 8.3, 8.4, 8.5

## Acceptance Criteria

### AC1 — Worker Success retorna schema valido
```
Given worker (ex: variety) executa com sucesso
When retorna WorkerResult
Then result eh instancia de WorkerSuccess (discriminated union)
And result.score eh int entre 0 e 100
And result.dimension_status == "ok"
And result.failure_reason eh None
And Pydantic validation passa sem exception
```

### AC2 — Worker Failure retorna schema valido
```
Given worker encontra erro durante analise
When retorna WorkerResult
Then result eh instancia de WorkerFailure
And result.score eh None (nao 0, nao 50)
And result.dimension_status in ("failed", "skipped", "insufficient_data")
And result.failure_reason eh string nao-vazia com contexto
And Pydantic validation passa sem exception
```

### AC3 — Worker crash e capturado como Failure
```
Given worker levanta exception nao-tratada
When app.py captura no pipeline
Then grava WorkerFailure implicito (score=None, status="crashed")
And failure_reason contem tipo da exception + mensagem
And NUNCA grava score=0 fake
```

### AC4 — DB schema refletedad verdade
```
Given WorkerSuccess com score=87
When evaluation_repo.save_analysis_result()
Then analysis_results.score == 87
And analysis_results.dimension_status == 'ok'
And analysis_results.failure_reason IS NULL

Given WorkerFailure
When evaluation_repo.save_analysis_result()
Then analysis_results.score IS NULL
And analysis_results.dimension_status == 'failed'
And analysis_results.failure_reason IS NOT NULL
```

### AC5 — Feature flag permite rollback
```
Given TRUTH_CONTRACT_ENABLED=false no env
When pipeline roda
Then comportamento legacy mantido (dict solto, score=0 em fallback)
And zero workers usam WorkerResult schema
And aggregator usa codepath antigo
```

### AC6 — Variety worker piloto migrado end-to-end
```
Given TRUTH_CONTRACT_ENABLED=true
When pipeline processa video e variety_analyzer roda
Then variety retorna WorkerResult valido
And DB grava novos campos corretamente
And aggregator le WorkerResult do variety sem quebrar os outros 9 workers (legacy)
And coexistencia worker-novo + workers-legacy funciona (schema tolerante)
```

### AC7 — Migration reversivel
```
Given migration aplicada
When executado rollback
Then colunas novas removidas
And dados legacy continuam intactos
And pipeline volta a funcionar com schema antigo
```

## Dev Notes

### Arquitetura Pydantic

Discriminated Union e o padrao idiomatico Pydantic v2 pra tipos alternativos baseados em campo. Exemplo:

```python
from typing import Annotated, Literal, Union
from pydantic import BaseModel, Field

class WorkerSuccess(BaseModel):
    model_config = {"extra": "forbid"}
    dimension: Literal["posture", "gesture", ...]
    dimension_status: Literal["ok"]
    score: int = Field(ge=0, le=100)
    metrics: dict
    confidence: float = Field(ge=0, le=1)

class WorkerFailure(BaseModel):
    model_config = {"extra": "forbid"}
    dimension: Literal["posture", "gesture", ...]
    dimension_status: Literal["failed", "skipped", "insufficient_data", "crashed"]
    failure_reason: str = Field(min_length=1)
    score: None = None  # explicit None, nao opcional

WorkerResult = Annotated[
    Union[WorkerSuccess, WorkerFailure],
    Field(discriminator="dimension_status")
]
```

Pydantic valida automaticamente pelo campo `dimension_status` e constroi a classe correta.

### Por que variety como piloto?

- Menor dependencia externa (so Whisper transcription + calculos numericos)
- Usado em teste automatizado de pipeline (existe fixture)
- Baixo risco se bugar (nao compoe score visivel diretamente — ponderado)
- Codigo relativamente limpo (vs identity_analyzer que ja tem retorno fake)

### O que NAO esta em 8.1

- Report generator migration (Story 8.5)
- Dashboards de saude do pipeline (Story 8.3)
- Retry + fallback automatico (Story 8.2)
